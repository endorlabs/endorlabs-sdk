# Non-Customer-Facing Resources

> **Internal Resources**: This document describes resources that are not intended for direct customer use in the Endor Cockpit SDK.

## Overview

Some resources in the Endor Labs API are implemented in the SDK for internal operations, policy evaluation, or system integration purposes. These resources are not meant to be directly manipulated by customers using the SDK.

## Non-Customer-Facing Resources

### DependencyMetadata

**Status**: Internal / System-Managed  
**Purpose**: Represents dependency relationships between PackageVersions  
**Use Case**: Used internally by the system to track dependency graphs and relationships  
**Customer Access**: Read-only, primarily used for policy evaluation

**Notes**:
- DependencyMetadata represents the relationship between an importer PackageVersion and a dependency PackageVersion
- All DependencyMetadata records are children of the root PackageVersion
- Used by policies to query dependency relationships and reachability analysis
- Customers should not create or modify DependencyMetadata directly

### LinterResult

**Status**: Internal / Scan-Generated  
**Purpose**: Contains results from third-party scanning tools (Gitleaks, Semgrep, etc.)  
**Use Case**: System-generated scan results that inform Findings  
**Customer Access**: Read-only, generated automatically by scans

**Notes**:
- LinterResult resources are automatically created during security scans
- Has RepositoryVersion or PackageVersion as parent
- Results are used to generate Findings automatically
- Customers should not create or modify LinterResult directly

### Metric

**Status**: Internal / Analytics-Generated  
**Purpose**: Contains output from analytics processing  
**Use Case**: System-generated metrics and scores (e.g., package_version_scorecard)  
**Customer Access**: Read-only, generated automatically by analytics

**Notes**:
- Metric resources are automatically generated by the analytics engine
- Has PackageVersion, RepositoryVersion, or Repository as parent
- Used extensively in policy evaluation (e.g., `package_version_scorecard`)
- Customers should not create or modify Metric directly

### Scan

**Status**: Internal / System-Managed  
**Purpose**: Represents scan execution and status  
**Use Case**: System-managed scan lifecycle tracking  
**Customer Access**: Read-only, managed by the scanning infrastructure

**Notes**:
- Scan resources are created automatically when scans are initiated
- Used to track scan progress and results
- Customers should not create or modify Scan directly

### ScanResult

**Status**: Internal / Scan-Generated  
**Purpose**: Contains scan results including findings, policies triggered, and scan metadata  
**Use Case**: System-generated scan results from endorctl or agentless scans  
**Customer Access**: Read-only, generated automatically by scans

**Notes**:
- ScanResult resources are automatically created during security scans
- Contains scan configuration, environment details, runtime statistics, findings, and error logs
- Typically generated by endorctl or agentless scanning infrastructure
- Most fields are system-generated and read-only
- Customers should not create or modify ScanResult directly (though API supports CRUD operations)

### Token (APIKey)

**Status**: Internal / Administrative  
**Purpose**: API key resource objects for programmatic access  
**Use Case**: API key lifecycle management (create, list, delete API keys)  
**Customer Access**: Administrative interfaces only (UI or API)

**Notes**:
- **Token/APIKey resource** (this API entity) is different from:
  - **API Credentials** (`ENDOR_API_CREDENTIALS_KEY`/`SECRET`): Long-lived key/secret 
    pair obtained when creating an API key, used for authentication
  - **ENDOR_TOKEN**: Short-lived authentication token obtained by exchanging 
    credentials with `/v1/auth/api-key` endpoint, used in Authorization headers
- Token/APIKey resources represent API key objects that can be created, listed, 
  and deleted through the API or UI
- The API credentials (`ENDOR_API_CREDENTIALS_KEY`/`SECRET`) are what you use 
  to authenticate with Endor APIs, not the Token resource itself

### Installation

**Status**: Internal / System-Managed  
**Purpose**: Represents installation/integration configurations  
**Use Case**: Platform integration management  
**Customer Access**: Administrative only

**Notes**:
- Installation resources track platform integrations (GitHub, GitLab, etc.)
- Managed through platform integration workflows
- Customers should not create or modify Installation directly

## Customer-Facing Resources

The following resources are intended for direct customer use:

- **Namespace** - Tenant organization and hierarchy
- **Project** - Project management and organization
- **Finding** - Security findings and vulnerabilities
- **Policy** - Custom policy creation and management
- **PackageVersion** - Package version information (replaces deprecated Package)
- **Repository** - Repository management (read-only, platform-synchronized)
- **RepositoryVersion** - Repository version information (read-only, platform-synchronized)
- **ScanResult** - Scan results and metadata (read-only, scan-generated)

## Deprecated Resources

### Package

**Status**: Deprecated  
**Replacement**: PackageVersion  
**Migration**: Use PackageVersion instead of Package resource

**Notes**:
- The Package resource has been deprecated in favor of PackageVersion
- All Package references should be updated to use PackageVersion
- PackageVersion provides version-specific package information and is the correct resource to use
